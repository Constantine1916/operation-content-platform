# 网站 H5 适配设计

日期：2026-04-18

## 背景

当前站点基于 Next.js App Router，整体布局明显偏桌面优先。已登录区域使用固定左侧栏和固定顶栏，顶栏通过硬编码 `left: 16rem` 与侧栏对齐；多个页面依赖宽屏网格、固定间距、较大的字号和 hover 才可见的操作信息。在手机视口下，这些假设会带来横向溢出、内容压缩、顶栏遮挡、操作难以触达和弹层可读性下降等问题。

本次工作只做 H5 适配，不改任何业务逻辑和功能。允许对移动端导航形态和页面排布做必要重构，但不改接口、权限、状态流、条件分支、字段结构和交互语义。

## 目标

1. 让全站所有可访问前台页面在常见手机浏览器中可正常浏览和操作。
2. 保持桌面端现有视觉语言和使用方式，移动端仅做必要的重排与触屏友好调整。
3. 将改动限制在布局、样式、展示层交互壳层，不触碰业务数据和功能逻辑。

## 非目标

1. 不新增业务功能，不删除已有功能。
2. 不调整接口协议、数据库字段、鉴权规则、分页策略、收藏逻辑、上传逻辑或生成逻辑。
3. 不做面向移动端的新信息架构或新内容模块。
4. 不引入新的设计系统或大规模视觉重做。

## 设计结论

采用“全局壳层重构 + 页面级响应式补齐”的方案。

原因：

1. 现有全局壳层包含明确的桌面假设，单纯靠零散断点修补不足以覆盖顶栏偏移、侧栏占位和安全区问题。
2. 单独维护一套 Mobile Layout 会显著提高后续维护成本，也超出“只做适配”的范围。
3. 通过先调整全局布局骨架，再逐页补齐内容容器、网格和弹层，可以在不改变业务逻辑的前提下，把风险收敛在表现层。

## 现状问题归类

### 1. 全局壳层问题

1. `MainLayout` 以 `h-screen`、固定侧栏和固定顶栏为核心，移动端虽然已有抽屉开关状态，但顶栏仍按桌面宽度偏移。
2. `Navbar` 使用内联样式固定 `left: 16rem` 和 `width: calc(100vw - 16rem)`，在手机宽度下不成立。
3. 顶栏、内容区和抽屉的滚动关系没有完全按移动端优先组织，存在头部压住内容和操作区的问题。

### 2. 页面布局问题

1. 首页使用大字号、固定宽度和多段大留白，窄屏下信息密度和首屏高度控制不足。
2. 概览、资讯、文章、图库、视频等页面存在多列网格或 Masonry 列数偏高的问题。
3. 登录注册、公开个人主页和个人中心部分区域虽然已部分响应，但仍有桌面尺寸、横向滚动和信息压缩风险。
4. `generate-img`、`md2image`、个人中心子面板等工具页和高密度页面更依赖桌面宽度。

### 3. 交互与弹层问题

1. 若干图片和视频卡片将信息与操作放在 hover 层，手机端无 hover，关键信息触达性不足。
2. 图片预览灯箱存在侧边信息栏布局，在手机端需要改为纵向堆叠或底部信息区。
3. 筛选器、标签栏和工具操作区在窄屏下容易换行失控或挤压主内容。

## 适配原则

1. 桌面端保持现有结构和视觉方向，移动端只做必要的导航重构与重排。
2. 以 CSS 断点和表现层组件改造为主，不改业务逻辑、接口调用和状态含义。
3. 以“可用性优先”而非“像桌面一样”为原则，允许双列改单列、侧栏改抽屉、信息侧栏改上下结构。
4. 所有关键信息和关键操作在触屏设备上不能依赖 hover 才能完成。
5. 先处理全局布局，再处理页面内容，最后处理弹层和细节状态。

## 模块设计

### 一、全局响应式壳层

涉及文件：

1. `components/MainLayout.tsx`
2. `components/Navbar.tsx`
3. `components/Sidebar.tsx`
4. `components/AuthLayout.tsx`
5. `app/globals.css`

设计：

1. 已登录区域保留桌面左侧导航模式，但在移动端改为顶部菜单按钮触发的抽屉式侧栏。
2. 顶栏改为默认全宽布局，不再依赖内联 `left/width` 偏移；桌面端通过布局容器自然避开侧栏。
3. 内容区统一改为移动优先的内边距和顶部留白，解决固定头部遮挡问题。
4. 为移动端补充安全区处理、滚动平滑和必要的 `overflow-x` 防护，避免出现页面级横向滚动。
5. 抽屉打开时仅处理 UI 层的遮罩、滚动和关闭行为，不影响任何业务状态。

### 二、公共页面框架

涉及页面：

1. 首页 `app/page.tsx`
2. 登录页 `app/login/page.tsx`
3. 注册页 `app/register/page.tsx`
4. 公开个人主页 `app/profile/[username]/page.tsx`

设计：

1. 首页保留当前视觉语言，但移动端收窄左右留白，降低部分标题与数字字号，降低首屏和大段 section 的垂直预算。
2. 多列统计与权益卡片在手机端改为纵向堆叠或两列以内。
3. 顶部导航按钮保持现有功能，仅调整间距和排布，确保小屏可点击。
4. 登录注册页继续维持中心卡片式结构，但在手机端确保表单、说明和跳转操作不超宽。
5. 公开个人主页的头部信息、标签栏、资源网格和空状态都按手机优先重排。

### 三、登录后内容页面

涉及页面：

1. `app/overview/page.tsx`
2. `app/hotspots/page.tsx`
3. `app/articles/page.tsx`
4. `app/ai-gallery/page.tsx`
5. `app/ai-video/page.tsx`

设计：

1. 页面头部统一使用响应式间距，避免顶栏和页头间距重复堆叠。
2. 概览页的统计卡片维持桌面多列，在手机端降为单列或双列，列表区继续纵向展示。
3. 热点与文章页的筛选条、平台切换和标签组允许横向滚动，不强制全部挤入一行。
4. 图库和视频瀑布流根据断点自动降列，手机端以单列或双列为主，保障卡片可读性和点击区域。
5. 若卡片中的说明、作者信息或操作当前依赖 hover，则在手机端调整为默认可见或稳定显示在底部内容区。

### 四、工具页与高密度页面

涉及页面：

1. `app/generate-img/page.tsx`
2. `app/md2image/page.tsx`
3. `app/profile/page.tsx`
4. `components/profile/*`

设计：

1. 这些页面保留当前功能流程、输入项、按钮语义和提交行为，只调整容器宽度、分栏方式和控件堆叠顺序。
2. 所有桌面多列控制区在手机端改为纵向或两列内布局。
3. 个人中心中的上传、资产、收藏、设置面板，保留 tab 逻辑和数据流，仅调整统计块、筛选条、资源网格和表单区域的排布。
4. 可横滑的 tab、标签和状态块允许保留横向滚动，但要保证外层不把整页撑出横向滚动。
5. 高风险区域优先使用“重排”而不是“重写”，避免引入逻辑回归。

### 五、弹层与预览组件

涉及文件：

1. `components/gallery/ImagePreviewLightbox.tsx`
2. 相关图片/视频/收藏卡片组件

设计：

1. 预览灯箱桌面端可保持图像与信息侧栏并排，手机端改为图像区域在上、信息区在下的纵向结构。
2. 关闭按钮、切换按钮、收藏按钮和复制按钮在手机端必须保持可见且可点击。
3. 所有依赖 hover 的遮罩层在手机端需要有可替代方案，优先采用默认显示关键说明和操作按钮的方式。
4. 不增加新功能，只改变内容暴露方式与布局顺序。

## 业务逻辑边界

以下内容明确保持不变：

1. 页面路由与访问控制逻辑。
2. 登录态读取与跳转逻辑。
3. 所有 API 请求路径、参数结构和响应消费方式。
4. 收藏、上传、分页、筛选、生成、复制、预览等功能逻辑。
5. 所有已有业务文案、状态文案和错误处理语义，除非某些文案因移动端布局需要做极小的显示层裁剪。

允许变化的内容：

1. DOM 结构中的布局容器层级。
2. Tailwind 类名、断点、间距、字号和显示隐藏策略。
3. 纯 UI 层本地状态，例如移动端抽屉开关、某些信息块的展开显示方式。

## 数据流与状态说明

本次适配不改变任何业务数据流。所有页面继续沿用现有的 `fetch`、Supabase session 获取、分页状态、收藏状态、上传状态与本地 UI 状态管理方式。

唯一允许新增的是与响应式体验直接相关的表现层状态，例如：

1. 移动端侧栏的开关状态。
2. 某些触屏信息区的展开状态。
3. 纯视觉层面的滚动锁定或遮罩状态。

这些状态不参与业务判断，不影响接口入参和业务结果。

## 错误处理

1. 保持现有 API 错误提示、空状态和加载状态语义不变。
2. 若因移动端重排导致某些状态组件需要换位置，只调整摆放，不改变触发时机和提示内容。
3. 对抽屉、灯箱和弹层增加更稳妥的关闭路径，保证在手机端误触成本可控。

## 验证计划

### 静态验证

1. 运行 lint 或等价静态检查。
2. 运行构建，确认没有类型或打包错误。

### 交互验证

1. 按手机视口逐页检查首页、登录注册、概览、资讯、文章、AI 图片、AI 视频、生成图片、MD 转图片、个人中心、公开个人主页。
2. 检查登录前、登录后、空状态、列表状态、弹层状态。
3. 检查移动端侧栏打开、关闭、切路由后的表现。
4. 检查灯箱、收藏按钮、复制按钮、筛选条、横向标签区和分页加载在手机端是否可用。

### 回归重点

1. `generate-img`
2. `md2image`
3. 个人中心上传/资产/收藏/设置
4. 图片预览灯箱
5. Masonry 列表页面

## 验收标准

1. 全站前台页面在常见手机宽度下无整页横向滚动。
2. 顶栏、抽屉、弹层和页面内容不存在互相遮挡导致无法操作的情况。
3. 所有现有功能可在手机端完成，不因缺少 hover 而失效。
4. 桌面端现有主要布局与功能行为保持稳定。
5. 代码改动集中在样式、布局与展示层组件，不涉及业务接口和逻辑变更。

## 风险与控制

### 风险

1. 高密度工具页在手机端可能出现局部控件挤压。
2. Masonry 与图片/视频卡片的 hover 行为在手机端需要平衡信息展示和视觉整洁。
3. 固定头部、抽屉、灯箱和滚动锁定之间可能出现交互冲突。

### 控制策略

1. 优先调整布局骨架，再处理各页面内部细节。
2. 优先做断点重排，不更换数据结构和事件流。
3. 对高风险页面逐页验证，不以全局 CSS “一把梭” 的方式处理。

## 决策摘要

本次 H5 适配以“移动端可用、功能不变、桌面端稳定”为核心目标。实现方式是先修正全局布局壳层，再逐页补齐响应式布局和触屏交互细节；任何改动都必须限制在表现层，不进入业务逻辑层。
